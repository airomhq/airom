AIROM's core scan is offline and deterministic: it reports what an artifact *is*. The
**CVE overlay** adds what is *known about it today* — it matches the AI package
dependencies AIROM already inventoried (by their [purl](https://github.com/package-url/purl-spec))
against the live [OSV.dev](https://osv.dev) advisory database and attaches the resulting
CVEs, with real CVSS v3 base scores computed from each vector, to those components.

<Warning>
**On by default — disable with `--no-cve` or `--offline`.** The overlay touches the
network, and it is **not deterministic across time**: the same scan surfaces more CVEs next
month as OSV grows. That is right for a vulnerability check and wrong for a reproducible
bill of materials, so turn it off (`--no-cve`) when you need a byte-stable BOM, or
`--offline` to also assert no network for the whole run.
</Warning>

<Info>
**Scope: AI dependencies, not a general-purpose SCA.** The overlay queries only the
components AIROM inventories — the AI/ML frameworks, SDKs, and serving libraries it
already identifies (pypi, npm, golang, cargo, maven, nuget purls). It answers "do the AI
parts of my stack have known CVEs?", not "audit my entire dependency tree." Use a
dedicated SCA for the latter.
</Info>

## Usage

```bash
airom fs .                                          # CVEs included by default
airom fs . -o cyclonedx=airom.cdx.json              # CVEs ride in vulnerabilities[]
airom fs . --exit-code 1 --fail-on "cve:high"       # fail CI on a high/critical CVE
airom fs . --no-cve                                 # skip the overlay (offline, byte-stable)
airom fs . --offline                                # skip it (and assert no network at all)
```

The overlay composes with everything else — a `--compliance` control that maps to "known
vulnerabilities" sees the CVEs (it runs before compliance), and every output format
projects them.

## How CVEs appear in output

<Tabs>
  <Tab title="CycloneDX">
    Each CVE becomes a top-level `vulnerabilities[]` entry with the CVE `id`,
    `source.name: osv.dev`, a `ratings[]` entry with `method: CVSSv31` carrying the **real
    score, severity, and vector**, aliases as `references[]`, and `affects[].ref` pointing
    at the component's `bom-ref`. The first fixed version rides in an
    `airom:cve.fixedVersion` property.

    ```bash title="CycloneDX with CVEs in vulnerabilities[]"
    airom fs . -o cyclonedx=airom.cdx.json
    ```
  </Tab>
  <Tab title="SARIF">
    Each CVE becomes a `cve/<id>` rule carrying GitHub's `security-severity` — the **real
    CVSS base score** here, not the synthetic marker the risk rules use — and a result
    (`error` / `warning` / `note` by severity) anchored to the manifest line that declared
    the vulnerable package.

    ```bash title="SARIF for code scanning"
    airom fs . -o sarif=airom.sarif
    ```
  </Tab>
  <Tab title="Native JSON / YAML">
    Each component gains a `vulnerabilities[]` array of
    `{id, aliases, severity, score, vector, summary, fixedVersion, source, url}`.
  </Tab>
  <Tab title="Table">
    The component table gains a `VULN` column (top severity + count, e.g. `high (2)`), the
    summary panel gains a `Vulnerabilities` breakdown by severity, and a per-CVE detail
    table follows — `LIBRARY / VULNERABILITY / SEVERITY / STATUS / INSTALLED / FIXED /
    TITLE`, most-severe first, with the advisory title and URL wrapped in the last column.
    Per-package columns (`LIBRARY`, `INSTALLED`, `FIXED`) merge vertically across a
    package's CVEs, Trivy-style, so the name and versions show once and span the group.
  </Tab>
</Tabs>

## Severity and the `--fail-on` threshold

Each CVE's severity comes from its CVSS v3.x vector — AIROM computes the base score per
the [FIRST formula](https://www.first.org/cvss/v3.1/specification-document) and buckets it
into the standard bands:

| Bucket | CVSS base score |
|---|---|
| `critical` | ≥ 9.0 |
| `high` | ≥ 7.0 |
| `medium` | ≥ 4.0 |
| `low` | > 0 |
| `unknown` | no parseable CVSS v3 vector (v2/v4-only or text-only advisory — vector shown, no score invented) |

`cve:<severity>` in `--fail-on` is a **threshold, not an exact match**:

```bash
airom fs . --fail-on "cve"           # any CVE at all
airom fs . --fail-on "cve:high"      # high AND critical
airom fs . --fail-on "cve:medium"    # medium and above
```

## Honesty and degradation

<Warning>
**Absence of CVEs is not a safety claim.** It means "OSV had no advisory for these exact
package versions at scan time," not "this dependency is safe." Advisories are published
continuously — re-run to re-check.
</Warning>

- **A network failure is never fatal — with one deliberate exception.** If OSV is
  unreachable, the affected component simply carries no CVEs and a warning lands in the
  scan's `Stats.Warnings` (visible under `--stats`); the AIBOM is never held hostage to a
  third-party API's uptime. **But when a CVE gate is active** (`--fail-on cve…`) and a
  component could not be checked, the scan **fails closed** (exit 2) instead of reporting a
  false "clean" — a gate that silently passes during an outage is worse than no gate.
- **Deduplication.** OSV often returns several records (a GHSA *and* a PYSEC, say) aliasing
  one CVE. AIROM collapses them to a single entry per id — keeping the most severe rating,
  preferring a real fixed version over a commit hash, and unioning the aliases — so you see
  one row per CVE, not one per advisory database.
