AIROM is a single static binary — `CGO_ENABLED=0`, no runtime dependencies, no
shared libraries to chase.

<Note>
  **AIROM is at `v0.1.6`** — the AI-native risk overlay, an opt-in CVE overlay
  (OSV.dev), compliance framework mapping, and a reworked terminal table. Early but real; expect rough edges. The `airom` command is on
  [PyPI](https://pypi.org/project/airom/), and cosign-signed static binaries for
  every supported platform ship on the [releases
  page](https://github.com/airomhq/airom/releases) with each tagged release. A
  Homebrew tap is planned.
</Note>

## pip

The easiest path — **no Go toolchain required**. The wheel bundles the static
`airom` binary *and* installs the Python SDK, so one command gives you both the
CLI and `import airom`. Requires **Python 3.10 or newer**.

```bash title="Install the CLI + SDK"
pip install airom
```

Prefer an isolated install that is always on your `PATH`? Use
[pipx](https://pipx.pypa.io):

```bash title="Isolated install"
pipx install airom
```

pip places `airom` on your `PATH` automatically inside an active virtualenv;
pipx does so globally. Then `airom --version` works from any directory. Wheels
are published for macOS (Intel + Apple Silicon), Linux (x86-64 + arm64, glibc +
musl), and Windows (x86-64).

## go install

Requires **Go 1.25 or newer** — reach for this if you already have a Go
toolchain.

```bash title="Install"
go install github.com/airomhq/airom/cmd/airom@latest
```

`@latest` resolves to the newest release tag (`v0.1.6` today).

### Put it on your PATH

This is the single most common installation problem, so handle it up front.

`go install` places the binary in `$(go env GOPATH)/bin`. If that directory is
not on your `PATH`, running `airom` gives you `command not found` — the install
succeeded, your shell just cannot see it.

Check where it went:

```bash title="Find the install directory"
go env GOPATH
```

Then add `bin` under that path to your shell profile:

<CodeGroup>
```bash title="zsh"
echo 'export PATH="$(go env GOPATH)/bin:$PATH"' >> ~/.zshrc
source ~/.zshrc
```

```bash title="bash"
echo 'export PATH="$(go env GOPATH)/bin:$PATH"' >> ~/.bashrc
source ~/.bashrc
```

```fish title="fish"
fish_add_path (go env GOPATH)/bin
```
</CodeGroup>

<Note>
  If you have set `GOBIN`, the binary goes there instead of `$(go env
  GOPATH)/bin`. Check with `go env GOBIN` and put that directory on your `PATH`.
</Note>

## Prebuilt binaries

Every tagged release publishes cosign-signed, `CGO_ENABLED=0` static binaries for
all supported platforms on the [releases
page](https://github.com/airomhq/airom/releases), each with a checksum and its own
SBOM. Download, verify, and drop the binary on your `PATH` — no runtime, no shared
libraries to install.

## Build from source

Clone and build. The `build` target produces a static, version-stamped binary at
`./airom` in the repo root.

<Steps>
  <Step title="Clone">
    ```bash title="Clone"
    git clone https://github.com/airomhq/airom.git
    cd airom
    ```
  </Step>

  <Step title="Build">
    ```bash title="Build"
    make build
    ```

    This runs a `CGO_ENABLED=0` build with `-trimpath` and stamps the version,
    commit, and build date into the binary. The result is `./airom` — run it in
    place, or move it somewhere on your `PATH`.
  </Step>

  <Step title="Or install to GOBIN">
    ```bash title="Install with the same version stamp"
    make install
    ```

    Same static build and version stamp as `make build`, but installed into
    `GOBIN` rather than left in the working tree.
  </Step>
</Steps>

<Tip>
  `make help` lists every target — the build, test, lint, and codegen entry
  points are all self-documenting.
</Tip>

### Overriding the version stamp

The version defaults to `git describe`, falling back to `v0.1.0-dev`. Override
it from the environment when you need a specific stamp:

```bash title="Custom version stamp"
make build VERSION=v0.1.0
```

## Verify the install

```bash title="Verify"
airom version
```

What you see depends on how you installed:

<Tabs>
  <Tab title="pip / pipx">
    ```text
    airom 0.1.2
      commit: 513e251
      built:  2026-07-17T18:31:13+05:30
      go:     go1.26.5 (darwin/arm64)
    ```

    The wheel ships a fully version-stamped binary, so a pip or pipx install
    reports the exact release it came from. (The SDK carries its own version —
    `pip show airom` — which can lead the tagged CLI release.)
  </Tab>

  <Tab title="go install">
    ```text
    airom dev
      commit: none
      built:  unknown
      go:     go1.26.5 (darwin/arm64)
    ```

    The build metadata is injected at build time with ldflags. A plain `go
    install` does not set it, so the binary honestly reports itself as an
    unstamped development build. The scanner is fully functional.
  </Tab>

  <Tab title="make build">
    ```text
    airom 0fb8a93-dirty
      commit: 0fb8a93
      built:  2026-07-17T06:20:56Z
      go:     go1.26.5 (darwin/arm64)
    ```

    `make` stamps the version from `git describe --tags --always --dirty`. With
    no tag in the repo yet, that resolves to the short commit SHA — plus
    `-dirty` if your working tree has uncommitted changes.
  </Tab>
</Tabs>

Either way, if `airom version` prints at all, the install worked. This is
exactly the `ToolInfo` block AIROM embeds in every AIBOM it writes, which is
what lets a document be traced back to the build that produced it.

<Accordion title="airom: command not found">
  It installed fine; your shell cannot find it. Where it went depends on how you
  installed:

  - **pip** — the binary is in your environment's `bin/`, on `PATH` automatically
    inside an active virtualenv. Outside one, activate the environment (or use
    `pipx`, which is always global). `pip show -f airom` lists the installed files.
  - **pipx** — always on `PATH`; if not, run `pipx ensurepath` and restart your shell.
  - **go install** — run `go env GOPATH`, confirm `$(go env GOPATH)/bin/airom`
    exists, and add that directory to your `PATH` (see [Put it on your
    PATH](#put-it-on-your-path) above).
  - **make build** — the binary is at `./airom` in the repo root and is not on your
    `PATH` at all until you move it.
</Accordion>
